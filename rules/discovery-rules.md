# Discovery Rules

Active during: `/wm:discover`, exploration, decision-making conversations.

1. **Present options driven by analysis.** The number of options equals the number of genuinely defensible paths — one, two, three, or more. Never pad to a fixed count. Every option must earn its place: if an option is obvious filler (weak "accept/defer", duplicates another option's trade-off, or exists only to fill a slot), drop it. When there's only one sensible path, say so — "Recommend X. No other option is defensible because Y" — don't invent alternatives. Scale presentation to decision weight.

2. **State trade-offs and lead with a recommendation.** When recommending, say what you'd lose with each choice. If trade-off analysis points to a clear winner, lead with the recommendation and name what was rejected — don't leave two or more equal options standing when one is clearly better. A one-option answer is valid when only one path survives the trade-off check; say so explicitly rather than padding to keep a menu shape.

3. **Treat uncertainty as a first-class input to discovery depth.** Distinguish "I know this" from "I'm inferring." Uncertainty is not just a conversational courtesy — it is an input to how much discovery is needed. High uncertainty expands discovery; low uncertainty contracts it. Flag uncertainty explicitly, then use it to size the next round of questions and the depth of the gray-area sweep.

4. **Challenge the solution, not the problem statement.** Don't agree by default. When the user states a problem and a proposed solution together, extract and stabilize the problem first, then challenge any proposed solution against it. Challenging the problem statement before it is stabilized wastes discovery cycles and can derail a valid pain point. Ordering: constraints → problem → solution. Never skip ahead.

5. **Extract constraints before exploring options.** Before proposing directions, surface what the user considers non-negotiable: time budget / appetite, non-goals, hard technical constraints, integration points, compliance rules. Write them down explicitly. Without this, discovery drifts into blue-sky brainstorms that ignore the real decision boundaries — and any options you generate get thrown away once the constraints finally surface.

6. **Separate problem from solution.** When the user proposes a solution, ask two questions before engaging with the proposal: "What problem does this solve?" and "Have we considered other ways to solve it?" Don't dismiss the proposed solution — use it as a data point about the user's model. The goal is to make the problem statement explicit and stable before committing to any one solution direction.

7. **Scale discovery depth to ambiguity and novelty, not stated size.** "Small" and "big" project labels are unreliable inputs to discovery depth. A "tiny fix" with an unknown root cause needs more discovery than a "big feature" with a clear spec. Size discovery to ambiguity and novelty, not scope labels. If the user calls something small but analysis shows high uncertainty, say so explicitly: "this is smaller in scope but higher in uncertainty — we should invest more discovery here than the label suggests."

8. **Dig into the hard parts.** Focus discovery on technical implementation, edge cases, concerns, and trade-offs — skip obvious questions and focus on what the user might not have considered. Use plain language to explain technical concepts. Before a decision point, give brief context first — don't jump from a question straight to an options list. Message shape follows comms rule 11 (skim test).
